Webcam Project Phase1

 

Webcam recognition

The first step in the research of webcams is to accurately identify webcams and their models among a large number of IPs, which is also the key to vulnerability detection. Port-based identification is very effective in most cases. The port is used as a service mark and can be used as a criterion for judging whether to run a certain service. At the same time, different types of devices often have certain characteristics due to their service characteristics. Therefore, port characteristics can be used as an important reference for device identification. RTSP (Real-time Streaming Protocol) service is a real-time multimedia streaming protocol used to control sound or video. It can establish and control one or more time-synchronized continuous streaming media, but it does not particularly emphasize time synchronization requirements, so it is more capable Tolerant of network delay, usually used for network camera equipment, its service port is generally 554. For a large number of known web camera devices, using nmap scanning software can find that most of the web camera devices have opened port 554 and run RTSP services. Therefore, port 554 and the corresponding RTSP service can be used as a fingerprint feature of a general-purpose webcam device. As long as port 554 is turned on, the device can be considered as a webcam device.

Information of the webcam used in our experiments

In essence, a webcam is an embedded system with video streaming function which is often an embedded Linux operating system. The webcam's related functions are realized by building a server and running related applications. The webcam used in our experiments, ieGeek 2.0 Mega Pixels IP Camera, can be seen used both Linux 2.6.32 and Linux 3.10 by Nmap scanning.

There are many ways for webcams to provide video services, and different connection methods use different ports.

A classic way is to use a web interface to display video images on a web page. Similar to the process of setting up a router, devices on the same network segment can enter the IP address of the camera in the browser address bar to realize the use and setting of the camera. In this way, we can obtain a lot of information about our equipment and set the equipment (default value and can be changed):

  • User name and password: admin/admin

  • Network port:

    • HTTP Port number: 80
    • RTSP Port: 554
    • RTMP Port: 1935 (PTSP permission check: off)

  • Remote access:

    • Main DDNS and 3th DDNS: off
    • uPnP Port Forwarding: off

  • Onvif: on

    • Server Port: 8080
    • Purview: No check
    • Time and Image Settings: Prohibit
    • NVC Type: Normal

  • P2P: Enable and UID: AACC-264381-AEFFD

  • More device information and system log

Most of these initial settings about the network can only be seen and modified in this way. However, the current webcam provides a quick access method, and customers generally do not enter this page when using it. Most of the existing client applications can scan the network and find a P2P-enabled webcam, then connect it within a few seconds.

The user manual device used in this experiment only provides how to use the default mobile app, CamHiPro, to connect and use the camera. And these mobile apps including several commonly used webcam apps downloaded from the Google Store often only provide simple image settings. Therefore, users may unknowingly use some functional protocols like Onvif that have vulnerability. In the experiment, although the apps provided in the user manual require account and password connections, some apps can still connect directly to the same network without a password through the Onvif function of the webcam.

At the same time, we see that the camera has a weak password. The initial default passwords of webcams are very simple and you can find the initial passwords of these manufacturers online, most of which are admin/admin.

The initial state of our device does not open remote access and uPnP, we will also open these two functions in the experiment for research.

Traffic capture

Since the previous process of using Wireshark on windows and various windows packet capture software will always be accompanied by various problems, we replaced all the experimental computer systems with Ubuntu Linux.

We first use a computer as a router to launch a hot spot and connect the webcam to this network, so that we can get all the traffic of the webcam. After that, use Wireshark to obtain the data traffic of the camera by filtering the IP address and MAC address of the webcam under different circumstances.

In order to study all the service protocols and ports provided by the webcam, in addition to the web connection method mentioned above, we also used the mobile phone application CamHipro provided in the user manual, and the PC software HiP2P Client provided by the ieGeek official website. The above several clients provide HTTP, TCP, RTMP and UDP transmission methods. At the same time, the popular tinyCam in the Google Store has also been used. It not only has many functions for viewing and modifying device information, but also supports P2P and Onvif. It can switch freely between TCP-based RTSP and UDP-based RTSP. The cloud service provided by tinyCam pro version can be used as follow-up research.

  • Data Preprocessing

    • Using Wireshark to export data with csv file directly

    • Format converting

      The raw data captured from Wireshark is in .pcap or .pcapng form, thus the first step is to extract the information into .csv (or whatever that is easy to deal with) form. Although Wireshark can output .csv file, this .csv has little information for "info" column, and we only use it to do simple analysis relevant to packet time.

      We use scapy module for python to extract information in "info" column. rdpcap function in scapy.all module can read packets into clear and accessible form shown below (This is a TCP packet).

      <Ether dst=ac:e0:10:16:d0:7f src=7c:a7:b0:b8:68:7a type=IPv4 |<IP version=4 ihl=5 tos=0x0 len=52 id=24745 flags=DF frag=0 ttl=64 proto=6 chksum=0xbe6c src=10.42.0.22 dst=118.190.154.176 |<TCP sport=55266 dport=30019 seq=3847346592 ack=3323370011 dataofs=8 reserved=0 flags=A window=1825 chksum=0xdbbe urgptr=0 options=[('NOP', None), ('NOP', None), ('Timestamp', (18921813, 596454881))] |>>>

      Since different packets have different structure, we have to divide a whole capture file into different types of packets, and of course, doing this may lose the relation between different types of packets.

    • Feature selection

      There are 21 features extracted from a packet. After statistical analysis, some columns are useless with std of 0, like IP version, and IP proto, since we have separated different types of packets in advance.

      Finally (for UDP packets) we choose only 9 columns as features, time, length of IP part, IP id, IP ttl, IP chksum, source port, destination port , length of UDP part, and UDP chksum.

  • Data analysis with python

    • Packet length

      We have analyzed the periodicity of UDP packets captured when webcam is connected to the network only, with no other device. First of all, we find the length of UDP packets varies in a small set, with cardinal of only 14.

      And since the numbers of packets of length 329, 342 are almost 0, we think them as outliners. After removing these abnormal packets, we get the box plot of packets length.

      It can be inferred from the graph that the length of most packets is between 40 to 70, and almost no packet has length above 100.

    • Periodicity

      By observing the raw data, we have found that there seems a pattern in traffic. The webcam will send packets to 3 addresses about every 60 seconds. This can be seen in I/O graph provided in Wireshark.

      To test our observation, we use DTW algorithm to check if the packets follow this pattern, and to find the period. DTW algorithm estimates the similarity of 2 time series by calculating the distance of them. We divide the traffic into a lot of time series, each with length "interval". Then we estimates the similarity of adjacent time series and sum them up. If we find the correct period, this sum will achieve its minimum.

      The result is shown in the figure.

      It is very clear that the period of this pattern is 60.

    • Other features

      As introduced in the last section, we have finally chosen 9 columns as our features. Actually, we can divide these features into 2 teams.

      The first team consists of "ip_len", "ip_ttl", "udp_len", and ip_len is always equal to udp_len+20. Features in this team vary in a small set, with cardinal of 10. They are somehow fixed for normal traffic. The figure shows the distribution of all possible udp_len.

      The features in the other team show continuity. They vary in a wide range and can be almost every value. The box plot is shown below.

      It is not surprising that check sum for IP and UDP show perfect symmetry, since they are hash values of information in a packet.

      In conclusion, features for UDP packets are very few, and only packet times and length of packets can be used in recognition for abnormal traffic.
